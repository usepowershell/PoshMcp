# PoshMcp v0.9.3 Release Notes

**Released**: 2026-05-01

## Security Fix

### Authentication Bypass — Configuration Precedence Override (CVE-Pending)

**Severity**: High

A second authentication bypass bug was discovered in the v0.9.3 release cycle, distinct from the v0.9.2 fix. The bug stems from how `WebApplicationBuilder` (the ASP.NET Core host builder) loads configuration files in order of precedence.

**Root Cause**: `WebApplicationBuilder` loads a baked-in base `appsettings.json` from the container image *before* adding the user's custom config file via `AddJsonFile()`. The base file contains `Authentication.Enabled: false` as a default. When the custom config (with `Authentication.Enabled: true`) is added later, the base file wins the configuration precedence — the custom value is silently overridden. All three authentication enforcement gates in `Program.cs` checked `authConfigValue.Enabled`, which resolved to `false` at runtime.

**Why v0.9.2 Didn't Catch This**: The v0.9.2 fix addressed a separate bug: `IOptions<AuthenticationConfiguration>` not being registered in the dependency injection container. That was a config binding problem. This bug is a config loading order problem. Both could have independently caused authentication bypass; v0.9.2 fixed the DI binding path, but the loading order problem remained in the middleware gate path.

**False Sense of Security**: Diagnostic tools (the `doctor` command and `get-configuration-troubleshooting` function) use `ConfigurationLoader.BuildRootConfiguration(finalConfigPath)` — a fresh `ConfigurationBuilder` seeded with only the custom config file and environment variables. They never load the base file. This created a dangerous divergence: the diagnostic tools correctly reported `Authentication.Enabled: true`, but the runtime enforced `Authentication.Enabled: false`.

**Impact**: Any container deployment with `Authentication.Enabled: true` in the custom config was affected. The HTTP middleware that should have enforced token validation was bypassed due to the configuration precedence override. All three authentication gates in `Program.cs` resolved the config value to `false`:
1. Early-return guard before middleware setup
2. `app.UseAuthentication()` and `app.UseAuthorization()` middleware registration
3. `mcpEndpoint.RequireAuthorization("McpAccess")` policy enforcement

Stdio and CLI transports are unaffected (they have no HTTP authentication middleware).

**Fix**: All three auth call sites in `Program.cs` now use a dedicated `authRootConfig` built via `ConfigurationLoader.BuildRootConfiguration(finalConfigPath)` — the same source the diagnostic tools use. This ensures complete parity: what `doctor` or `get-configuration-troubleshooting` reports is exactly what the runtime enforces.

```csharp
// All three auth gates now use the same configuration source
var authRootConfig = ConfigurationLoader.BuildRootConfiguration(finalConfigPath);
var authConfig = authRootConfig.GetSection("Authentication").Get<AuthenticationConfiguration>();
```

## Diagnostic Parity

The v0.9.3 fix also closes the diagnostic parity gap. Previously:
- Running `poshmcp doctor` or calling `get-configuration-troubleshooting` might report `Authentication.Enabled: true`
- But the runtime would enforce `Authentication.Enabled: false`

After v0.9.3, these are guaranteed to match. If the diagnostic tools say auth is enabled, the runtime will enforce it.

## Breaking Changes

None.

## Upgrade Notes

### For Existing Deployments

**This is a security fix and all deployed instances should be updated immediately.**

1. **Rebuild and redeploy your container image** from v0.9.3
2. **No configuration changes needed** — your existing `appsettings.json` with `Authentication.Enabled: true` will now work correctly
3. **Verify enforcement** — after upgrade, token validation will now be enforced if you have authentication configured
4. **Run diagnostics** — use `poshmcp doctor` or call `get-configuration-troubleshooting` to confirm authentication is now both reported and enforced correctly

### Deployments Without Authentication

If your deployment has `Authentication.Enabled: false` or does not configure an authentication section, this fix has no impact — requests continue to be accepted without authentication.

## Testing

This release includes:
- New regression tests verifying that `authRootConfig` correctly reflects values from the custom config file
- Tests confirming that all three auth gates now resolve to the same value
- Tests verifying diagnostic parity: `doctor` output matches runtime enforcement
