# PoshMcp v0.9.2 Release Notes

**Released**: 2026-05-01

## Security Fix

### Authentication Configuration Not Applied (CVE-Pending)

**Severity**: High

A critical bug in `AddPoshMcpAuthentication()` prevented authentication configuration from being applied to the dependency injection container. The method called `.Get<AuthenticationConfiguration>()` to read the configuration but never registered it with `services.Configure<AuthenticationConfiguration>()`. This caused `IOptions<AuthenticationConfiguration>` to always resolve to its default state (`Enabled = false`), regardless of what was specified in `appsettings.json`.

**Impact**: Any deployment with `Authentication.Enabled: true` configured was affected. The HTTP middleware that should have enforced token validation was never activated. All MCP requests were accepted without authentication, even when a JWT or API key requirement was configured.

Two security gates in `Program.cs` were bypassed:
1. `app.UseAuthentication()` and `app.UseAuthorization()` middleware were not registered
2. `mcpEndpoint.RequireAuthorization("McpAccess")` policy was not applied

**Fix**: The configuration registration now occurs unconditionally in `AddPoshMcpAuthentication()` before any early-return guards:

```csharp
services.Configure<AuthenticationConfiguration>(configuration.GetSection("Authentication"));
```

This ensures `IOptions<AuthenticationConfiguration>` always reflects the current configuration, and authentication middleware is correctly activated when enabled.

## Breaking Changes

None.

## Upgrade Notes

### For Existing Deployments

**This is a security fix and all deployed instances should be updated immediately.**

1. **Update your PoshMcp deployment** to version 0.9.2 or later
2. **Redeploy all instances** — updates to container images, binaries, or running services are required
3. **No configuration changes needed** — existing `appsettings.json` with `Authentication.Enabled: true` will now work correctly
4. **Verify activation** — after upgrade, token validation will now be enforced if you have authentication configured

### Deployments Without Authentication

If your deployment has `Authentication.Enabled: false` or does not configure an authentication section, this fix has no impact — requests continue to be accepted without authentication.

## Testing

This release includes:
- 3 new regression tests verifying that `IOptions<AuthenticationConfiguration>` correctly reflects values from `appsettings.json`
- Tests confirming that authentication middleware is activated when `Enabled: true`
- Tests confirming that `RequireAuthorization()` is applied to protected endpoints
