# PoshMcp v0.9.7 Release Notes

**Released**: 2026-05-02

## Overview

This release fixes duplicate entries in the `/.well-known/oauth-protected-resource` (PRM)
response — specifically `scopes_supported`, `bearer_methods_supported`, and
`authorization_servers` appearing more than once. The duplicates were caused by two
`IOptions<AuthenticationConfiguration>` Configure registrations both calling Bind on the
same object without clearing list properties first.

## What's Fixed

### Duplicate Entries in Protected Resource Metadata (PRM) Response

**Symptom**: `GET /.well-known/oauth-protected-resource` returned arrays with duplicate values:

```json
{
  "scopes_supported": [
    "api://my-app/user_impersonation",
    "api://my-app/user_impersonation"
  ],
  "bearer_methods_supported": ["header", "header", "header"]
}
```

**Root Cause**: Two `IOptions<AuthenticationConfiguration>` Configure registrations were
appending (not replacing) list properties:

1. `RunHttpTransportServerAsync` registered `.Configure(opts => authRootConfig.GetSection("Authentication").Bind(opts))`
2. `AddPoshMcpAuthentication` independently called `services.Configure<AuthenticationConfiguration>(configuration.GetSection("Authentication"))`

Both delegates operated on the same options instance (not a fresh copy), and since
`IConfigurationSection.Bind` appends to existing `List<T>` properties without clearing them
first, each item was added once per Configure registration. The `BearerMethodsSupported`
property also had a non-empty default initializer (`= new() { "header" }`), adding a third
"header" entry when two registrations each appended their value.

**Fix**:
- Removed the duplicate `services.Configure<AuthenticationConfiguration>` call from
  `AddPoshMcpAuthentication` — the caller already registers this via the fluent Options API.
- Changed `ProtectedResourceConfiguration.BearerMethodsSupported` default initializer from
  `new() { "header" }` to `new()` (empty). The `ProtectedResourceMetadataEndpoint` already
  falls back to `["header"]` when the list is empty.

**Impact**: PRM responses now contain exactly one entry per value. MCP clients that rely on
these arrays for authorization server discovery will no longer encounter unexpected duplicates.

## Upgrade Notes

No configuration changes required. If you are generating PRM responses from a
`ProtectedResourceConfiguration` object created without binding from config, note that
`BearerMethodsSupported` now defaults to an empty list. The endpoint itself falls back to
`["header"]` when the list is empty — behavior is unchanged for HTTP clients. If you are
constructing `ProtectedResourceConfiguration` in code and relying on the default value,
explicitly set `BearerMethodsSupported = ["header"]`.

## Compatibility

- No breaking changes from v0.9.6
- All v0.9.5/v0.9.6 features remain unchanged
