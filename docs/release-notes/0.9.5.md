# PoshMcp v0.9.5 Release Notes

**Released**: 2026-05-01

## Overview

This release adds OAuth Authorization Server (AS) proxy and Dynamic Client Registration (DCR) proxy endpoints, enabling MCP clients to authenticate against Entra ID without requiring users to manually provide a `client_id` or prompting errors when clients attempt to register dynamically. With these new endpoints, any MCP client (VS Code, Claude Desktop, Cline, or custom clients) can now complete the OAuth 2.1 + PKCE flow transparently.

## What's New

### OAuth Authorization Server Proxy (RFC 8414)

**Feature**: New `/.well-known/oauth-authorization-server` endpoint advertising Entra ID's real authorization and token endpoints.

**Details**:
- Endpoint responds with RFC 8414-compliant AS metadata document
- Advertises Entra ID's `authorization_endpoint` and `token_endpoint` (pointing to `login.microsoftonline.com`)
- Advertises this server's `/register` endpoint as the DCR registration endpoint
- Includes supported scopes: `openid`, `profile`, `email`, `offline_access`, and your configured Audience scope (e.g., `api://poshmcp-prod/.default`)
- Reports support for `authorization_code` grant and `S256` PKCE
- Honors Azure Container Apps' `X-Forwarded-Proto` and `X-Forwarded-Host` headers to construct correct absolute URLs

This endpoint is always anonymous (`.AllowAnonymous()`) — metadata is public by design and does not require authentication.

### Dynamic Client Registration (DCR) Proxy (RFC 7591)

**Feature**: New `POST /register` endpoint that returns a pre-configured `client_id` without requiring Entra ID DCR.

**Details**:
- Accepts POST requests to `/register` (standard DCR endpoint location)
- Returns HTTP 201 Created with JSON payload:
  ```json
  {
    "client_id": "your-configured-client-id",
    "client_id_issued_at": 1714602000,
    "token_endpoint_auth_method": "none"
  }
  ```
- Entra ID does not natively support RFC 7591 DCR for public clients; this proxy removes the need for users to manually paste a `client_id` into their MCP client
- Endpoint is always anonymous — no authentication required to fetch metadata

### Protected Resource Metadata (PRM) Auto-Population

**Feature**: When OAuth proxy is enabled, the Protected Resource Metadata endpoint now auto-populates `authorization_servers` at request time if the configuration list is empty.

**Details**:
- Previously, you had to manually configure `ProtectedResource.AuthorizationServers` with your server's base URL
- Now, if `OAuthProxy.Enabled: true` and `AuthorizationServers` is empty or unset, the PRM automatically populates the list with the server's base URL (honoring `X-Forwarded-*` headers)
- This simplifies deployment — fewer config keys to set

## What's Fixed

### MCP Clients Unable to Auto-Register

**Symptom**: MCP clients (e.g., Claude Desktop, Cline, custom clients) prompted users to manually enter a `client_id` or failed with errors when attempting to fetch the DCR endpoint.

**Root Cause**: 
- Entra ID does not support RFC 7591 Dynamic Client Registration for public clients (only confidential clients with a client secret)
- When PoshMcp's OAuth metadata did not advertise a registration endpoint, clients with no pre-registered `client_id` either:
  1. Prompted users to paste the `client_id` (bad UX)
  2. Attempted to call a non-existent endpoint and failed silently

**Fix**: PoshMcp now acts as a lightweight DCR proxy:
1. `/.well-known/oauth-authorization-server` advertises `registration_endpoint: {server}/register`
2. When clients POST to `/register`, the server returns the configured `client_id` immediately
3. Clients proceed without user intervention

**Impact**: All MCP clients (VS Code, Claude Desktop, Cline, etc.) now work seamlessly. VS Code's hardcoded client_id continues to work; other clients get the configured client_id automatically via the proxy.

## Configuration

To enable the OAuth proxy, set the following environment variables (or equivalent in `appsettings.json`):

```bash
# Required to enable the proxy
Authentication__OAuthProxy__Enabled=true

# Azure/Entra ID tenant ID (GUID)
Authentication__OAuthProxy__TenantId=12345678-1234-1234-1234-123456789012

# Client ID to return from /register endpoint
# This MUST be pre-authorized in your Entra ID app registration
Authentication__OAuthProxy__ClientId=aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee

# Application ID URI for your server (audience in token validation)
# Example: api://poshmcp-prod
Authentication__OAuthProxy__Audience=api://poshmcp-prod
```

### Equivalent `appsettings.json`

```json
{
  "Authentication": {
    "OAuthProxy": {
      "Enabled": true,
      "TenantId": "12345678-1234-1234-1234-123456789012",
      "ClientId": "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee",
      "Audience": "api://poshmcp-prod"
    }
  }
}
```

## Upgrade Notes

### For Existing Deployments with Authentication Enabled

If you have `Authentication.Enabled: true` and want to support MCP clients without pre-registered credentials:

1. **Create or identify your Entra ID app registration** (or use the same one from v0.9.4)
2. **Get the Client ID** from your app registration (Application / Client ID field)
3. **Set the four new environment variables** (as shown in Configuration above):
   - `Authentication__OAuthProxy__Enabled=true`
   - `Authentication__OAuthProxy__TenantId={your-tenant-id}`
   - `Authentication__OAuthProxy__ClientId={your-client-id}`
   - `Authentication__OAuthProxy__Audience={your-audience-uri}`
4. **Pre-authorize the client_id in Entra ID**:
   - Go to your app registration in Azure Portal
   - Navigate to **Expose an API** → **Authorized client applications**
   - Add the Client ID you're returning from `/register`
   - Also add VS Code's built-in client_id if you're using VS Code: `aebc6443-996d-45c2-90f0-388ff96faa56`
5. **Clear or update `ProtectedResource.AuthorizationServers`**:
   - If already set, you can leave it as-is or clear it
   - If empty, it will auto-populate from the OAuth proxy when requests arrive
6. **Rebuild and redeploy your container image**
7. **Verify OAuth flow**: Test with an MCP client (VS Code, Claude Desktop, etc.)

### For Deployments Without Authentication

No changes needed. If `Authentication.Enabled: false` or the authentication section is not configured, the OAuth proxy is not registered and has no impact.

### For Deployments Using Managed Identity

If you're using Azure Managed Identity for server credentials and App Registration for OAuth scopes:
1. Follow the same steps as above
2. Ensure the Managed Identity has permissions to call Azure APIs if needed
3. The OAuth proxy works alongside Managed Identity without conflict

## Compatibility

- Requires PoshMcp v0.9.5 or later
- Works with any MCP client that supports RFC 8414 (OAuth 2.0 Authorization Server Metadata) discovery
- Tested with:
  - VS Code (MCP extension)
  - Claude Desktop (MCP integration)
  - Cline (MCP support)
  - Any RFC 8414-compliant client
- Entra ID tenant required (Office 365, Microsoft 365, Azure Active Directory)
- No changes to existing token validation logic or security posture
