# PoshMcp v0.9.12 Release Notes

## Bug Fixes

### Fix: Protected Resource Metadata — duplicate scopes and invalid `resource` URI

Two issues in the `/.well-known/oauth-protected-resource` (PRM) endpoint were causing VS Code and other MCP clients to log `AggregateError: Failed to fetch resource metadata from all attempted URLs` during OAuth discovery:

**1. Duplicate array values in PRM response**

When `Authentication.Enabled: true` was configured, `scopes_supported` and `bearer_methods_supported` arrays in the PRM response contained each value twice (e.g., `["user_impersonation", "user_impersonation"]`).

*Root cause:* `AuthenticationConfiguration` options were bound twice in `HttpServerHost` — once directly and once via `AddPoshMcpAuthentication`. Each binding appended values to the existing lists, doubling every entry.

*Fix:* Removed the redundant `AddOptions<AuthenticationConfiguration>().Configure()` registration in `HttpServerHost.cs`. Only `AddPoshMcpAuthentication` now performs the binding.

**2. Non-HTTPS `resource` URI breaks VS Code PRM validation**

RFC 9728 requires the `resource` field in PRM to be an HTTPS URI that clients can use to validate the metadata document round-trip. When `ProtectedResource.Resource` was set to an Entra Application ID URI (`api://...`), VS Code attempted to fetch `api://.../.well-known/oauth-protected-resource` as an HTTP request, which always fails.

*Fix:* `ProtectedResourceMetadataEndpoint` now substitutes the server's canonical HTTPS base URL (derived from `X-Forwarded-Proto`/`X-Forwarded-Host` headers) as the `resource` field when:
- The OAuth proxy is enabled (`OAuthProxy.Enabled: true`), AND
- The configured `Resource` value does not start with `https://`

The Application ID URI (`api://...`) continues to be used as the JWT `aud` audience for token validation — only the PRM `resource` field changes.
