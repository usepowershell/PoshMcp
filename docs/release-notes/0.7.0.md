# PoshMcp v0.7.0 Release Notes

**Released**: 2026-04-01

## What's New

### Stdio Logging to File

PoshMcp now suppresses diagnostic logs from the stdio stream to prevent corruption of the MCP JSON-RPC protocol. Instead, logs are written to a configurable file, enabling reliable operation in stdio transport mode while maintaining full diagnostic visibility.

**Three configuration methods (priority: CLI > environment > appsettings):**

- **CLI flag** — `poshmcp serve --log-file /path/to/logfile.log`
- **Environment variable** — `POSHMCP_LOG_FILE=/path/to/logfile.log`
- **Configuration file** — Set `Logging.File.Path` in `appsettings.json`

**Example with environment variable:**

```bash
export POSHMCP_LOG_FILE=/var/log/poshmcp.log
poshmcp serve --transport stdio
```

**Example with appsettings.json:**

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information"
    },
    "File": {
      "Path": "/var/log/poshmcp.log"
    }
  }
}
```

When running in stdio mode without a log file configured, the server runs silently — no logs are written anywhere, preserving the JSON-RPC stream integrity.

### MimeType Nullable in Resources

The `MimeType` field in MCP Resources is now optional. When not specified, it defaults to `text/plain` to ensure MCP client compatibility.

**Updated example:**

```json
{
  "McpResources": {
    "Resources": [
      {
        "Uri": "poshmcp://resources/readme",
        "Name": "Readme",
        "Description": "Project documentation",
        "Source": "file",
        "Path": "README.md"
      }
    ]
  }
}
```

If needed, you can still explicitly set `MimeType` for other content types (e.g., `application/json`).

## Configuration

### File Logging

The file logging feature writes all diagnostic logs to disk without polluting the stdio stream. This is essential for:

- **Debugging** — Capture logs during stdio MCP server operations
- **Production monitoring** — Persistent audit trail independent of MCP client
- **Container deployments** — Volume-mount the log file for log aggregation

**Docker example with persistent logs:**

```bash
docker run -d \
  -v /host/logs:/data \
  -e POSHMCP_LOG_FILE=/data/poshmcp.log \
  -e POSHMCP_TRANSPORT=stdio \
  poshmcp:latest
```

## Breaking Changes

None. This is a backward-compatible enhancement.

## Bug Fixes & Stability

- Fixed stdio stream corruption caused by diagnostic logs being written during MCP server operation
- Improved resource handling to safely handle nullable MimeType values
- Enhanced error logging for troubleshooting without affecting JSON-RPC protocol integrity

## Upgrade Notes

### For Existing Deployments

If you are using stdio transport mode:
- Upgrade to 0.7.0 to enable file logging (highly recommended for production)
- Configure `POSHMCP_LOG_FILE` or use `--log-file` to capture diagnostic logs
- Existing configurations remain compatible

If you are using HTTP transport mode:
- No changes required; logs continue to write to console
- Safe to upgrade immediately

### Recommended Next Steps

1. If using stdio mode, configure file logging for production deployments:
   ```bash
   poshmcp serve --transport stdio --log-file /var/log/poshmcp.log
   ```
2. For Docker deployments, add volume mounting to persist logs:
   ```bash
   docker run -d -v /host/logs:/app/logs -e POSHMCP_LOG_FILE=/app/logs/poshmcp.log poshmcp:latest
   ```
3. Review [Transport Modes Guide](../articles/transport-modes.md) for detailed configuration

## Known Limitations

- Log files are not automatically rotated; implement external log rotation if needed (e.g., logrotate on Linux)
- File logging is stdio-mode only; HTTP mode continues to use console logging
- Log files must be writable by the process user; directory creation is not automatic

## Platform Support

- **Windows**: .NET 10 runtime
- **Linux/macOS**: .NET 10 runtime
- **Docker**: Included in `poshmcp:latest` image
- **Azure Container Apps**: Supported with volume mounting for log persistence
