# PoshMcp v0.9.4 Release Notes

**Released**: 2026-05-01

## Overview

This release fixes OAuth 2.1 discovery for VS Code MCP clients. Previously, VS Code could not complete the OAuth flow because PoshMcp's 401 challenge was missing the `resource_metadata` hint required by RFC 9728, causing VS Code to redirect to PoshMcp's own server instead of to Entra ID.

## Fixed

### VS Code OAuth Discovery (High)

**Symptom**: When using PoshMcp with VS Code and authentication enabled, VS Code redirected the browser to `https://{poshmcp-host}/authorize` instead of to Entra ID (`login.microsoftonline.com`). The OAuth flow could not complete.

**Root Cause**: The JWT Bearer 401 challenge was missing the `WWW-Authenticate: Bearer resource_metadata="..."` header required by RFC 9728 (OAuth 2.0 Protected Resource Metadata). Without it, VS Code cannot discover the Protected Resource Metadata endpoint and falls back to treating the resource server as the authorization server. Specifically:

1. VS Code receives a 401 response with only `WWW-Authenticate: Bearer` (no `resource_metadata` field)
2. VS Code cannot discover the Protected Resource Metadata (PRM) endpoint
3. VS Code assumes PoshMcp is the authorization server and constructs `{baseUrl}/authorize`
4. Request to a non-existent endpoint fails, OAuth flow breaks

**Fix**: The JwtBearer challenge handler in `AuthenticationServiceExtensions.cs` now emits the full RFC 9728-compliant `WWW-Authenticate` header:

```
WWW-Authenticate: Bearer resource_metadata="{scheme}://{host}/.well-known/oauth-protected-resource"
```

VS Code can now correctly:

1. Parse the `resource_metadata` URL from the challenge header
2. Discover the PRM endpoint and fetch the OAuth metadata
3. Read `authorization_servers` from the PRM to identify Entra ID
4. Redirect to Entra ID's `/authorize` endpoint (`login.microsoftonline.com`)
5. Complete the OAuth 2.1 + PKCE flow

**Impact**: Affects all deployments with `Authentication.Enabled: true` that use VS Code as the MCP client.

### ApiKey Handler Metadata URL (Low)

**Symptom**: The ApiKey authentication challenge constructed an invalid metadata URL using the `api://` URI scheme instead of `https://`.

**Root Cause**: The handler derived the Protected Resource Metadata URL from `ProtectedResource.Resource` (an `api://` URI), producing an invalid non-HTTP URL that clients cannot fetch.

**Fix**: Metadata URL is now derived from the incoming request's scheme and host, consistent with the JwtBearer fix:

```
resource_metadata="{Request.Scheme}://{Request.Host}/.well-known/oauth-protected-resource"
```

## Upgrade Notes

### For Existing Deployments

No configuration changes required. Update the container image or binary to v0.9.4 to pick up the fixes.

1. **Rebuild and redeploy your container image** from v0.9.4
2. **No configuration changes needed** — your existing `appsettings.json` will work correctly
3. **Verify discovery** — after upgrade, test VS Code OAuth flow by connecting PoshMcp in VS Code with authentication enabled
4. **Run diagnostics** — use `poshmcp doctor` to confirm the authentication configuration is correct

### Deployments Without Authentication

If your deployment has `Authentication.Enabled: false` or does not configure an authentication section, this fix has no impact — requests continue to be accepted without authentication.

## Compatibility

- Requires `.well-known/oauth-protected-resource` endpoint to be responding (included in PoshMcp since v0.9.x)
- Works with VS Code's built-in MCP OAuth client (PKCE/OAuth 2.1 flow)
- Works with any RFC 9728-compliant MCP client
- No changes to Entra ID app registration required
- RFC 9728 compliance ensures forward compatibility with future MCP clients

## Testing

This release includes:

- New integration tests verifying that 401 challenges include the correct `WWW-Authenticate` header
- Tests confirming VS Code can parse the `resource_metadata` URL
- Tests verifying ApiKey handler constructs valid HTTPS metadata URLs
- Tests validating the Protected Resource Metadata endpoint response
