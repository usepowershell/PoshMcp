# PoshMcp v0.9.18 Release Notes

## Bug Fixes

### Fix: JWT audience validation — accept multiple audience formats

Added `ValidAudiences` list to `AuthSchemeConfiguration` so the server can
accept tokens where the `aud` claim is either the Application ID URI
(`api://APP_ID`) or the bare GUID, depending on the token version issued by
Entra ID.

Previously, only the single `Audience` property was validated against, causing
`SecurityTokenInvalidAudienceException` when Entra issued a v1 token with a
different audience format than expected.

**Configuration change (for existing deployments):**
Add `ValidAudiences` to your `appsettings.json` Bearer scheme:

```json
"Schemes": {
  "Bearer": {
    "Type": "JwtBearer",
    "Authority": "https://login.microsoftonline.com/<tenantId>/v2.0",
    "Audience": "api://80939099-d811-4488-8333-83eb0409ed53",
    "ValidAudiences": [
      "api://80939099-d811-4488-8333-83eb0409ed53",
      "80939099-d811-4488-8333-83eb0409ed53"
    ]
  }
}
```

### Fix: Enhanced JWT failure diagnostics

`OnAuthenticationFailed` now decodes the incoming token (without signature
validation) and logs the actual `aud` and `iss` claims alongside the exception
type and message. This makes it immediately obvious when an issuer or audience
mismatch occurs, without needing to manually decode the token.

Log format:
```
JWT OnAuthenticationFailed: SecurityTokenInvalidAudienceException: ... — token aud=[api://APP_ID] iss=[https://sts.windows.net/TENANT/]
```
