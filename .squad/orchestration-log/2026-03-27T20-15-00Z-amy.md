## Work Summary

Successfully refactored Azure Bicep infrastructure to use module-based architecture for subscription-scoped deployment with zero breaking changes.

### Problem Addressed

Original `main.bicep` had 10 compilation errors preventing deployment:
- **BCP139** (×5): "A resource's scope must match the scope of the Bicep file"
- **BCP265** (×3): "`resourceGroup` is not a function"
- **BCP037** (×2): "Property 'scope' not allowed"
- **BCP120**: "Expression requires compile-time value"

Root cause: Attempted to deploy all resources at subscription scope while setting `scope: resourceGroup()` on individual resources, violating Bicep's scope rules.

### Solution Implemented

**Architecture:** Module-based cross-scope deployment
```
main.bicep (targetScope = 'subscription')
├── Creates Resource Group
├── Invokes resources.bicep module (scope: az.resourceGroup(rg.name))
│   └── Deploys Container Apps, Log Analytics, App Insights, Managed Identity
└── Assigns RBAC role at subscription level
```

**Key Changes:**
1. **main.bicep**: Subscription-scoped orchestrator, creates resource group, invokes module, assigns RBAC role, aggregates outputs
2. **resources.bicep**: Resource group-scoped resources module (no structural changes, already correct)
3. **Role Assignment**: Fixed GUID generation to use deterministic values (compile-time) instead of runtime principal ID

### Best Practices Demonstrated

1. **Modules for Cross-Scope Deployment** - Proper Bicep pattern for subscription → RG scenarios
2. **No `name` on Module** - Modern Bicep convention for single-module references
3. **Symbolic References** - Using `resources.outputs.*` instead of `resourceId()` functions
4. **`az.*` Namespace** - Correct function syntax for Azure-specific functions
5. **Deterministic GUIDs** - Role assignment names calculable at compile time
6. **Fail-Fast Validation** - Bicep compilation catches errors before deployment
7. **In-Place Updates** - Architecture supports upgrading existing deployments
